Data Vault, as it actually happened to you

01 · HUB Who
Memory hook: Hub = ID card. One row per real-world thing.
Holds
The unique business key (e.g. a claim ID) + a hashkey that identifies it everywhere else in the model.
Rule
A business key appears in the Hub exactly once, no matter how many times it shows up in source data.
02 · LINK How things relate
Memory hook: Link = a bridge. Connects two or more Hubs.
Holds
The relationship between business keys — e.g. a claim linked to a provider, or a claim linked to a remittance.
Rule
No descriptive data lives here — just which keys are connected.
03 · SATELLITE What changed, when
Memory hook: Satellite = the diary. Tracks history of a Hub or Link over time.
Holds
Descriptive/attribute columns (FILE_DATE, status, amounts) + a load date, so you can see how a record looked at any point in time.
Your incident
The Ascension gap traced back to NULL FILE_DATE values in exactly this kind of satellite — SAT_REMITTED_CLAIM_ATTR and SAT_REMITTED_SERVICE_ATTR. Root cause was upstream (a stuck watermark), but the symptom showed up here first.
04 · HASHDIFF The shortcut
Memory hook: Hashdiff = one zipper instead of twenty individual buttons.
What it is
A hash of all a satellite's descriptive columns, mashed together into one value.
Why
New row comes in → compare its hashdiff to the last one on file. Same hash = nothing changed, skip it. Different hash = insert a new dated row. One comparison instead of checking 20 columns by hand.
Not this
Hashdiff is not what makes a business key unique — that's the Hub's job. Hashdiff only detects change inside a Satellite.
Say this out loud before you walk in
HUB=Who — the unique key
LINK=How they relate — the bridge
SAT=What changed, when — the diary
HASH=The shortcut — one zipper, not 20 buttons
Anchor phrase: "A hashdiff tells my satellite whether anything changed, without me checking every column by hand — it's not about uniqueness, that's the Hub's job."
