Why Data Vault over 3NF or Star Schema
3NF (OLTP) Star Schema (Kimball) Data Vault
History usually overwritten on update — you lose what a record looked like yesterday unless you build custom audit tables. Handles history via Type 2 dimensions, but retrofitting a new "track this too" attribute means reworking the dimension. History is the default. Every satellite row is dated. Nothing is ever overwritten — you always get built-in history.
Tightly coupled — a schema change in one table can ripple into every query joining it. Fact/dimension grain is fixed early; a new source or new grain often means redesigning the star. Hubs/Links/Satellites load independently and in parallel. A new source just adds a new satellite — existing structures don't change.
Not built for auditability — hard to prove what the data looked like at a point in time for compliance. Better than 3NF, but conformed dimensions across many facts get complex to fully audit. Fully auditable by design — this is exactly why RCM/healthcare and other regulated industries lean on it.
Fast to query for a single transaction, painful for broad reporting. Built for reporting speed — this is Kimball's core strength. Slower to query raw — usually a Business Vault or reporting layer sits on top for performance.
Why this isn't theoretical for you: when the Ascension watermark bug hit, the damage was contained to two satellites (SAT_REMITTED_CLAIM_ATTR, SAT_REMITTED_SERVICE_ATTR). The Hubs and Links — the actual keys and relationships — were untouched. That's what made a targeted 19.4M-row backfill possible instead of a full reload. In a Star Schema, a bug like that usually means rebuilding the fact table itself, because the broken data and the keys live in the same row.
