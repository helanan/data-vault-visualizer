Notice: no raw column was ever "the Link." The Link isn't sourced from the file — it's built by combining Hub keys that showed up together on the same row. That's a common trip-up: Links model relationships, not attributes.
Why Data Vault over 3NF or Star Schema
3NF (OLTP) Star Schema (Kimball) Data Vault
History usually overwritten on update — you lose what a record looked like yesterday unless you build custom audit tables. Handles history via Type 2 dimensions, but retrofitting a new "track this too" attribute means reworking the dimension. History is the default. Every satellite row is dated. Nothing is ever overwritten.
Tightly coupled — a schema change in one table can ripple into every query joining it. Fact/dimension grain is fixed early; a new source or new grain often means redesigning the star. Hubs/Links/Satellites load independently and in parallel. A new source just adds a new satellite.
Not built for auditability — hard to prove what the data looked like at a point in time for compliance. Better than 3NF, but conformed dimensions across many facts get complex to fully audit. Fully auditable by design — why regulated industries (RCM, banking) lean on it.
Fast for a single transaction, painful for broad reporting. Built for reporting speed — Kimball's core strength. Slower to query raw — usually a Business Vault/reporting layer sits on top.
